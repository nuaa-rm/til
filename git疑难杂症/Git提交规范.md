# Git 提交 (Commit) 新手指南及规范

对于刚接触 Git 的初学者来说，了解如何正确地保存和记录代码更改是非常重要的一步。本文档将先介绍 `git commit` 的基础用法，然后再说明本项目推荐的提交信息格式规范。

---

## 🚀 第一部分：初识 `git commit`

### 1. 什么是 `git commit`？
你可以把 Git 仓库想象成一本可以随时存档的魔法书。当你修改了文件后，首先需要用 `git add` 将它们放入**暂存区**（准备存档的状态）。
接下来使用的 `git commit` 命令，就是正式将暂存区里的更改**打成一个包，并贴上一个标签（提交信息）**，永久保存到你的本地仓库历史中。

### 2. 常用的 `git commit` 参数

对于日常开发，你通常只需要掌握以下几个最常用的参数：

#### ① `git commit -m "提交信息"` 
**最常用的方式**。`-m` 代表 message。它允许你直接在终端里输入简短的提交说明，而不需要打开文本编辑器。
```bash
git add file.txt
git commit -m "新增了文件file.txt"
```

#### ② `git commit -a -m "提交信息"` (或 `-am`)
**快捷方式**。`-a` 代表 all。这个参数会自动把所有**已经被 Git 跟踪过**且被修改或删除的文件放入暂存区，并直接提交。
> **注意：** 它可以帮你省去 `git add` 这一步，但**不能**自动包含你全新创建（以前从未 commit 过）的未跟踪文件！
```bash
# 等价于先 git add 已修改的文件，再 git commit
git commit -am "修改了之前已存在的所有文件"
```

#### ③ `git commit --amend`
**“后悔药”**。当你刚刚完成了一次提交，突然发现：
- 哎呀，提交信息打错字了！
- 哎呀，少用 `git add` 漏掉了一个文件！
此时，你可以先 `git add` 漏掉的文件，然后运行 `git commit --amend`。它会将新的更改和上次的提交合并成一个新的提交，或者让你重新编辑上一次的提交信息。
```bash
git commit -m "修复了一个BUG"
# 发现漏掉了一个文件
git add missing_file.txt
git commit --amend -m "修复BUG并补充遗漏的文件"
```

#### ④ 仅仅输入 `git commit` (不带任何参数)
如果你直接敲击回车，Git 会打开你系统配置的默认文本编辑器（通常是 Vim 或 Nano）。这种方式适合当你需要编写一段**非常详细、包含多行解释**的提交信息时使用。

## 📝 第二部分：Git 提交规范

为了保持代码提交历史的清晰、可读和可维护，本项目建议在编写提交信息时遵循以下基于标签的格式。

### 1. 提交格式

建议的整体格式如下：
```
[标签] 简短描述

- 详细描述（可选）
- 影响范围（可选）
```

### 2. 标签定义

在简短描述前加上带有方括号的标签，能让人一眼看出这次提交属于什么性质的更改。

| 标签 | 说明 | 示例 |
| :--- | :--- | :--- |
| **[ADD]** | 新增功能、新文件或新资源 | `[ADD] 新增用户登录接口` |
| **[FIX]** | 修复 Bug 或解决已知问题 | `[FIX] 修复上传文件时的内存泄漏问题` |
| **[MOD]** | 修改现有逻辑、优化代码或更新配置 | `[MOD] 调整搜索算法的权重分配` |
| **[DEL]** | 删除多余的代码、文件或资源 | `[DEL] 移除过时的 API 调试脚本` |
| **[DOC]** | 更新文档（README, 接口文档, 注释等） | `[DOC] 完善 SSH 端口配置说明` |
| **[REF]** | 代码重构（不改变功能，仅优化结构） | `[REF] 抽离数据库连接池为独立模块` |
| **[TEST]** | 新增或修改测试用例 | `[TEST] 增加订单支付流程的单元测试` |
| **[CI]** | 持续集成相关的配置更改 (GitLab 等) | `[CI] 更新构建流水线以支持 Node 20` |
| **[CHORE]** | 其他琐事（如更新 .gitignore, 依赖包升级等） | `[CHORE] 升级 lodash 到最新版本` |

### 3. 编写原则

1. **简洁明了**：第一行（标题）最好控制在 50 个字符以内。
2. **使用动词**：建议标题部分的简短描述使用“新增”、“修复”、“优化”、“删除”等动词开头。
3. **原子性**：尽量保证一个 commit 只做一件事。如果修改了两个完全不想干的 Bug，请分成两次 commit。
4. **补充信息**：如果变动较复杂，请在使用默认编辑器（或 `-m` 参数加换行）时，在标题下方**留一空行**，然后编写详细的原因和影响范围。

### 4. 规范使用示例

**简单提交：**
```bash
git commit -m "[ADD] 实现局域网 ROS 禁用脚本"
git commit -m "[FIX] 修复脚本在 Ubuntu 24.04 下的兼容性问题"
```

**包含详细描述的复杂提交（使用多行信息）：**
```bash
git commit -m "[MOD] 优化数据库查询逻辑

- 将原有的嵌套循环查询改为批量IN查询
- 降低了大约30%的数据库 CPU 占用率
- 影响范围：用户订单列表接口"
```
